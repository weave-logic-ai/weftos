**Owner**: human + multi-agent swarm.
**Plane cycle**: `0.7.x` (UUID `e3df6167-3b59-46e4-bee8-7f37146b9a9f`).
**Source of truth**: every item is a Plane work item (`WEFT-N`); this doc
is a *plan view*, not a duplicate task list. Updates to status happen
in Plane via `scripts/plane.sh transition / close / defer`.
**Aggregate**: 148 items across 13 workstreams. Breakdown by finding:
security 34, docs 29, stub 19, bug 15, tests 13, tooling 13, tech-debt 9,
gap 6, orphan 4, governance 4, performance 2. Priorities: 1 urgent,
72 high, 46 medium, 29 low.

> **Cycles are gates, not time-boxed sprints** (CLAUDE.md / handoff
> rule). Items target the cycle they must ship by, not when they
> happen. Anything that slips out of 0.7.x must be deferred (cycle
> bump + comment) — not silently dropped.

## Topology

148 items don't run as 148 parallel tracks. They run as **6 milestones
in dependency order**, with **fan-out within each milestone**. Between
milestones is a hard gate: build-clean + tests-green + Plane
state-update before the next milestone starts.

## Execution mechanics

### Branch strategy

- Each milestone gets a tracking branch off `development-0.7.0`:
  `m1-truth-and-docs`, `m2-gateway-hardening`, `m3-channels`,
  `m4-multi-agent`, `m5-voice-cost-tests`, `m6-release-polish`.
- Each agent's PR within a milestone targets its milestone branch.
- At milestone gate, milestone branch merges into
  `development-0.7.0`. Next milestone branches off the new tip.
- Final cut: `development-0.7.0` → `master` for the 0.7.0 tag.

### Per-agent workflow (every agent does this)

```bash
# 1. Claim the Plane item
scripts/plane.sh transition <WEFT-id> "In Progress" --assignee me

# 2. Branch
git checkout -b weft-<N>-short-slug m<X>-<milestone>

# 3. Work

# 4. Gates
scripts/build.sh check
scripts/build.sh clippy
scripts/build.sh test    # or targeted -p <crate>

# 5. Commit + PR

# 6. Close after merge
scripts/plane.sh close <WEFT-id> \
  --shipped "..." \
  --commits <SHA1>,<SHA2> \
  --tests "..." \
  --build "scripts/build.sh check + clippy clean" \
  --followups <WEFT-...>  # if any
```

### Coordinator workflow (per milestone)

```bash
# 1. Filter Plane to just this milestone's items
scripts/plane.sh list-cycle 0.7.x | grep -E "ws05|ws07|..."

# 2. Spawn agents in parallel (max 8 to stay under Plane's 4 req/sec)
#    via Claude Code Task tool, one prompt per agent, in ONE message

# 3. As agents return, review their PRs, merge into milestone branch

# 4. At gate: scripts/build.sh gate on milestone branch
#    Plane: every milestone item in Done
#    Stamp the audit doc footer with closed item summary

# 5. Merge milestone branch to development-0.7.0; cut next milestone
```

### What does NOT happen this wave

- Items deferred to 0.8.x stay in 0.8.x (don't pull them forward).
- 1.0.x and 0.9.x items are not in scope.
- The 19-item agent-core-v1.1 backlog is mostly 0.8.x; only the cost
  circuit-breaker (WEFT-322) is in this wave.

### Failure modes & recovery

- **Agent stalls** (600s watchdog): re-dispatch with narrower scope
  (single workstream / single bucket). Document via Plane comment
  on the affected items.
- **Plane API 429 / 403**: wrapper retries on 429; on 403 (Cloudflare
  WAF), sanitize the description (remove literal shell command
  syntax) and retry just that item. See WEFT-550 for the established
  workaround.
- **Build break in a milestone**: revert the offending PR, re-open
  the affected Plane item with `transition <id> "Todo"` + comment
  explaining the regression.
- **Item turns out duplicate** (e.g. WEFT-298 vs WEFT-101): close
  both and create one new clean item, OR close the duplicate with
  `superseded-by-WEFT-N` comment. Don't ship the same fix twice.
